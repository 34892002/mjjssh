# 脚本与订阅设计

> 状态：已实现第一、二阶段
>
> 本文定义 MJJSSH 的脚本管理与 JSONL 订阅功能。目标是让用户在 SSH 主机上复用常用命令，并能够订阅可信的公开脚本集合。本文不代表脚本执行功能已经实现。

## 1. 目标与边界

脚本页面位于侧边栏“主机”“密钥”之后，提供两类脚本：

- **我的脚本**：用户在本地创建、编辑、复制、删除的脚本；作为 Vault 业务数据的一部分，可随现有加密云同步同步到用户设备。
- **订阅脚本**：从 HTTPS JSONL 地址下载的只读脚本集合；不写入 Vault 的业务同步数据，不会被上传到 GitHub Gist 或 Gitee 私有代码片段。

首版范围：

- 管理本地脚本。
- 添加、启用、禁用、刷新和删除订阅源。
- 浏览和搜索订阅脚本；订阅脚本可复制到“我的脚本”后编辑。
- 从当前已连接的 SSH 会话中选择脚本，预览并在用户明确确认后执行。

明确不包含：

- 后台定时执行、开机执行、自动触发或无需确认的执行。
- 下载并运行二进制文件、脚本自更新、插件机制或脚本内嵌凭据。
- 跨主机批量自动执行。
- 对任意脚本来源的“安全”保证。订阅源和脚本内容均应视为不可信输入。

## 2. 用户体验

### 2.1 脚本页

侧边栏新增“脚本”。页面使用“我的脚本”和“订阅”两个标签页：

- **我的脚本**：显示名称、描述、标签、最近修改时间；提供新建、编辑、复制、删除和执行入口。
- **订阅**：先展示订阅源列表及刷新状态；选中来源后显示脚本列表。脚本卡片显示来源名称、版本、描述、标签和风险提示。
- 两个列表均支持名称、描述和标签搜索。

执行入口可以出现在已连接会话的工具栏或脚本详情页。用户先选择目标会话，再显示不可编辑的命令预览；仅在点击“执行”后才通过该会话写入终端。

### 2.2 创建本地脚本

创建表单包含：

| 字段 | 必填 | 规则 |
|---|---|---|
| 名称 | 是 | 1-80 个字符，在本地脚本中唯一 |
| 描述 | 否 | 最多 500 个字符 |
| 标签 | 否 | 最多 10 个，每个最多 32 个字符 |
| 命令 | 是 | UTF-8 文本，最大 32 KiB，不得为空 |
| 风险级别 | 是 | `low`、`medium`、`high`，默认 `medium` |

命令保持原文，不由客户端改写、拼接或解释。多行 shell 脚本允许保存和执行，但 UI 必须清楚显示完整文本，不折叠危险命令。

### 2.3 订阅管理

“订阅”页提供“添加订阅”入口，填写名称与 HTTPS 地址。首次发布时内置且默认启用官方订阅：

```text
https://raw.githubusercontent.com/34892002/mjjssh/refs/heads/main/subs/sub_main.jsonl
```

官方订阅应显示为“官方”，但用户仍可禁用。订阅不应在应用启动时自动请求；用户点击刷新时下载。后续自动刷新若实现，必须默认关闭并显示最近更新时间、HTTP 状态和失败原因。

刷新成功后，应用解析、校验并缓存最新有效内容；刷新失败时继续使用上一次成功缓存，不清空已显示脚本。每个订阅脚本均可“复制到我的脚本”，副本解除只读限制且不再随订阅更新。

## 3. 本地数据模型

### 3.1 Vault 中的用户脚本

用户脚本是私有业务数据，在 `vault.json` 增加 `scripts` 数组，并随现有 Vault 云同步整体加密：

```json
{
  "formatVersion": 2,
  "scripts": [
    {
      "id": "0fc729a2-8d52-4e62-a940-7df57d73cc3e",
      "name": "查看磁盘使用率",
      "description": "显示主要文件系统的磁盘使用情况",
      "tags": ["system", "disk"],
      "command": "df -h",
      "riskLevel": "low",
      "createdAt": "2026-07-24T12:00:00Z",
      "updatedAt": "2026-07-24T12:00:00Z"
    }
  ]
}
```

字段规则：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | UUID | 本地脚本唯一标识 |
| `name` | string | 用户可见名称 |
| `description` | string / null | 简短用途说明 |
| `tags` | string[] | 用于筛选和搜索 |
| `command` | string | 原始 SSH shell 命令或多行脚本 |
| `riskLevel` | string | `low` / `medium` / `high`，仅用于提示，不构成安全判断 |
| `createdAt` | RFC3339 | 创建时间 |
| `updatedAt` | RFC3339 | 最近修改时间 |

不为脚本持久化主机、用户名、密码、私钥、Token 或环境变量。目标 SSH 会话只能在执行时由用户选择。

### 3.2 订阅配置与缓存

订阅配置与下载缓存不进入 Vault。它们保存在应用数据目录的独立文件中，例如：

```text
data/script-subscriptions.json
data/script-subscription-cache/<subscription-id>.jsonl
```

原因：订阅地址和缓存是设备级的外部内容；将其纳入 Vault 会导致私有云同步把第三方来源、缓存版本和可能无效的大文件同步到其他设备。

订阅配置格式：

```json
{
  "formatVersion": 1,
  "subscriptions": [
    {
      "id": "official",
      "name": "MJJSSH 官方脚本",
      "url": "https://raw.githubusercontent.com/34892002/mjjssh/refs/heads/main/subs/sub_main.jsonl",
      "enabled": true,
      "official": true,
      "etag": "\"abc123\"",
      "lastFetchedAt": "2026-07-24T12:00:00Z",
      "lastSuccessAt": "2026-07-24T12:00:00Z",
      "lastError": null
    }
  ]
}
```

使用 HTTP `ETag` 和 `If-None-Match` 减少重复下载；HTTP 304 表示缓存仍有效。缓存文件使用原始 JSONL 内容，以便后续版本重新解析并保留来源可追溯性。

## 4. JSONL 订阅协议

### 4.1 文件要求

- 内容类型建议为 `application/x-ndjson` 或 `application/jsonl`；客户端不依赖响应头判断。
- UTF-8 编码，允许 UTF-8 BOM。
- 每个非空行是一个独立 JSON 对象；空白行和以 `#` 开头的注释行忽略。
- 单个文件最大 1 MiB，单行最大 64 KiB，最多 1,000 条脚本。超出限制时拒绝更新并保留旧缓存。
- 协议版本由每条脚本的 `schemaVersion` 声明，首版为 `1`。

### 4.2 脚本记录

```json
{"schemaVersion":1,"id":"disk-usage","name":"查看磁盘使用率","description":"显示主要文件系统的磁盘使用情况","tags":["system","disk"],"command":"df -h","riskLevel":"low","platforms":["linux"],"version":"1.0.0"}
```

字段定义：

| 字段 | 必填 | 规则 |
|---|---|---|
| `schemaVersion` | 是 | 固定为整数 `1` |
| `id` | 是 | `[a-z0-9][a-z0-9-]{0,63}`；在同一订阅文件内唯一 |
| `name` | 是 | 1-80 个字符 |
| `description` | 否 | 最多 500 个字符 |
| `tags` | 否 | 最多 10 个标签，每个最多 32 个字符 |
| `command` | 是 | 非空 UTF-8 命令文本，最多 32 KiB |
| `riskLevel` | 是 | `low` / `medium` / `high` |
| `platforms` | 否 | `linux`、`macos`、`windows` 的非重复数组；缺省表示未声明适用系统 |
| `version` | 是 | 发布者定义的版本字符串，最多 64 个字符 |
| `homepage` | 否 | HTTPS URL，指向文档或源码 |

未知字段必须忽略，以支持协议向后兼容。字段类型错误、缺少必填字段、超过长度限制或重复 `id` 的记录应被跳过，同时在刷新结果中报告行号和原因；只要整个文件可下载且至少一条记录有效，更新即可成功。

### 4.3 来源与唯一性

订阅脚本的全局运行时标识为：

```text
<subscription-id>:<script-id>
```

不同订阅源允许使用相同 `id`，但在 UI 中必须显示来源。不得根据名称合并不同来源的脚本，也不得将更新后的订阅脚本静默覆盖用户复制出的本地脚本。

## 5. 下载与校验

1. 用户点击刷新，客户端仅接受 `https://` URL；拒绝 `http`、`file`、本地 IP、回环地址以及跳转到这些目标的重定向。
2. 设置连接、读取和总下载超时；首版建议分别为 10 秒、30 秒和 45 秒，重定向最多 3 次。
3. 发送 `Accept: application/x-ndjson, application/jsonl, text/plain` 以及已有缓存时的 `If-None-Match`。
4. 成功响应后，在内存中完成尺寸检查、JSONL 解析和字段校验；全部失败时不写入缓存。
5. 至少一条记录有效时，原子写入新缓存并更新 `lastSuccessAt`、`etag`。无效行作为警告显示，不阻止使用有效脚本。
6. 网络错误、非 2xx/304 状态、超限或无有效记录时记录错误，保留上一份成功缓存。

官方 URL 使用 `main` 分支，适合持续分发但不是不可变发布源。未来应支持发布者提供带版本 tag 或提交 SHA 的 URL，供需要可复现内容的用户固定版本。

## 6. 执行安全

脚本是远端 shell 命令，风险不能由客户端可靠判定。`riskLevel` 只是作者的声明，不能作为自动允许的依据。

执行规则：

- 永远不自动执行订阅或本地脚本。
- 每次执行都必须由用户选择已连接的 SSH 会话，并在确认对话框中查看名称、来源、完整命令和风险等级。
- `medium` 和 `high` 脚本默认要求额外确认；`high` 脚本要求用户输入脚本名称或点击明确的“仍要执行”。首版可统一要求一次确认，但不得取消预览。
- 命令通过当前 SSH 会话的既有输入通道写入终端；不创建隐藏会话，不绕过主机密钥校验，也不将输出上传给订阅源。
- 执行日志不保存命令正文、终端输出、密钥或连接凭据。诊断包同样不得包含这些内容。
- 订阅脚本可包含破坏性命令。官方标识表示来源归属，不代表适用于所有主机或环境。

状态：`已完成`

### 第三阶段：可选增强

- 订阅源签名或校验和，解决 HTTPS 来源被账户或分支更新后内容变化的问题。

- 订阅源版本固定、变更对比和用户确认升级。

## 8. 验收标准

- 用户可创建、编辑、复制、删除并执行本地脚本；脚本在现有 Vault 云同步后仍完整可用。
- 官方订阅默认出现，且可由用户禁用；刷新时仅请求配置的 HTTPS URL。
- 正确的 JSONL 文件能显示有效脚本；单条无效记录只产生警告，不影响其他有效记录。
- 下载失败或解析结果为空时，旧缓存仍可浏览和使用。
- 订阅脚本不可原地编辑；复制后成为独立的本地脚本，不再受订阅更新影响。
- 没有用户显式操作和确认时，不会通过 SSH 执行任何脚本。
- 订阅地址、脚本正文和终端输出不进入诊断包；订阅缓存不进入 Vault 云同步。
